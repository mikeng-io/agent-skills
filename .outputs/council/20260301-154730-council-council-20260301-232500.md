## Multi-Model Confirmed Findings

### MMC001 — HIGH — HALTED/SKIPPED Semantics Contract Broken
**DA-challenged: confirmed**
**Confirmed by:** claude, gemini, codex, opencode (4/4 bridges)

The HALTED vs SKIPPED status contract is inconsistently defined across the suite. bridge-commons defines HALTED as "requires user decision — wait; do not continue." bridge-codex and bridge-opencode intentionally return HALTED for auth/setup decisions. TOPOLOGY states bridges are "non-blocking" and unavailability produces SKIPPED. deep-council Step 5 overrides bridge-commons semantics in non-interactive mode by auto-SKIPping HALTED bridges — but this conversion policy exists only in deep-council prose and is absent from bridge-commons, TOPOLOGY, and the council-report-schema.json output fields.

**Evidence:** bridge-commons "HALTED vs SKIPPED" table; TOPOLOGY lines 25–26; bridge-codex Step 2; bridge-opencode Step 2; deep-council Step 5 non-interactive default; council-report-schema.json (no auto_skipped_halted_bridges field).

**Remediation:** Add the HALTED→SKIPPED conversion policy to bridge-commons as a documented orchestrator responsibility. Add `auto_skipped_halted_bridges` as a required field in the council report JSON schema and Step 7 template. Align TOPOLOGY to reflect that HALTED is valid at the bridge layer but converted at the orchestrator layer.

### MMC002 — HIGH — Stateless CLI Context Overflow
**DA-challenged: confirmed (scope narrowed to stateless paths)**
**Confirmed by:** claude, gemini (2/2 bridges that analyzed this; codex touched adjacent concerns)

CLI bridges (Gemini CLI, Codex CLI, OpenCode CLI) must embed all previous-round findings in Round N prompts. bridge-commons Context Passing Between Rounds specifies this explicitly for stateless paths but defines no size limit, summarization strategy, or file-based injection path. For large codebases or thorough intensity, accumulated context can exceed CLI character limits or model context windows with silent truncation — producing unreliable or partial challenge responses that are indistinguishable from valid partial responses in the output schema.

**Evidence:** bridge-commons "Context Passing Between Rounds" table; bridge-gemini Execution section; bridge-codex Step 3B; bridge-opencode Step 3B.

**Remediation:** Add a maximum context size constant to bridge-commons. Require stateless bridges to summarize prior-round findings above the threshold before embedding. Alternatively, write Round 1 outputs to a temp file and pass the file path in the Round 2 invocation.

## Single-Source Confirmed Findings

### SS002 — MEDIUM — Preflight Threshold Contradiction
**Source:** claude (also seen as related issue X005 in codex)

preflight/SKILL.md "When to Invoke" states: "A calling skill should invoke preflight when two or more of the following are true." deep-council Step 0 states: "Invoke Skill(preflight) only if context_report.confidence == low OR one or more of these signals remains unresolved." These are directly contradictory threshold definitions in co-authoritative documents. The discrepancy means deep-council may trigger preflight when preflight's own logic would self-terminate, propagating assumption-filled scope to all bridge Task prompts.

**Evidence:** preflight/SKILL.md "When to Invoke"; deep-council/SKILL.md Step 0 conditional.

**Remediation:** Align both documents to "one or more unresolved signals" (lower threshold is more conservative and matches caller intent).

### INT002 — HIGH — Context Truncation Inflates Multi-Model Confirmation Confidence
**Domain:** integration (MMC002 × Stage A deduplication threshold)

MMC002 (stateless CLI context overflow) and Stage A mechanical deduplication interact to create false confidence. When Round N prompts for stateless bridges grow large, LLM context windows silently drop early-round findings. Surviving findings are disproportionately those appearing near the end of the embedded context window. If two stateless CLI bridges both truncate to the same recent context window, they can "agree" on a finding not because it was independently confirmed, but because both truncated to the same survivor set. Stage A's 70% description overlap threshold treats this truncation-survivor agreement as multi-model confirmation with the same weight as genuine independent convergence.

**Evidence:** bridge-commons "Context Passing Between Rounds" (stateless paths, no size cap); deep-council Stage A ("Finding confirmed by 2+ bridge reports (>70% description overlap)"); bridge output schema (no context_length or truncation_detected field).

**Remediation:** Require bridges to report approximate prompt size per round. Add a `shared_cli_path: true/false` flag to Stage A merging logic — findings confirmed only by bridges sharing the same stateless CLI execution path should be flagged as lower-confidence confirmations.

## Synthesis Notes

All four bridges returned CONCERNS. No bridge returned FAIL — there are no CRITICAL findings in the suite. The architecture is coherent; the two-layer debate design, bridge-commons contract, and domain-registry routing are sound.

The most structurally significant finding is the HALTED/SKIPPED semantic inconsistency (MMC001), confirmed by all four bridges independently and reinforced by the INT001 integration finding. This pattern — where a status defined in bridge-commons is silently converted at the orchestrator layer without being reflected in the output schema — is the primary gap between the suite's stated guarantees and its actual runtime behavior.

The context overflow risk (MMC002 + INT002) is the second-tier concern: it affects the reliability of multi-model confirmation itself, which is the central value proposition of deep-council. INT002's finding that truncation-survivor agreement can inflate confirmation confidence is the sharpest finding to emerge from the cross-bridge synthesis layer and was not present in any single bridge report.

Three findings were withdrawn by DA challenge, including the session idempotency concern (SS003, which rested on a false premise about filename collisions) and the Debate Coordinator SPOF (SS006, which applies generically to all Task agent spawns). The DA withdrawal process demonstrates the suite's own debate architecture functioning as designed in this very audit.
