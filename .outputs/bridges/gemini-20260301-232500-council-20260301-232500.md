## Summary

The agent-skills suite exhibits strong architectural alignment but suffers from terminology duplication and inconsistent verdict thresholds between bridge and council layers. Significant risks identified in non-interactive error handling (HALTED bridge silencing) and context payload limits for stateless CLI bridges.

## Findings

### G001 — [HIGH] Missing 'audit' task_type in bridge-commons

**Domain:** api
**Type:** finding
**Severity:** HIGH

**Description:** The bridge-commons SKILL.md defines a task_type enum (review|planning|implementation|analysis|research) that excludes 'audit', despite deep-council identifying 'audit' as a primary intent in Step 0. This results in undefined framing and verdict logic when an audit task is dispatched to bridges.

**Evidence:** bridge-commons/SKILL.md section 'Input Schema'; deep-council/SKILL.md Step 0 'Merge into working scope'

**Action:** Add 'audit' to bridge-commons task_type enum and define its framing/verdict logic in the corresponding tables.

---

### G002 — [MEDIUM] Conflicting Output ID Prefixes

**Domain:** api
**Type:** finding
**Severity:** MEDIUM

**Description:** Prefix conventions are inconsistent across the suite. bridge-commons defines C/G/X/O, but debate-protocol uses 'F' for findings, and deep-council uses 'MMC' (multi-model-confirmed), 'SS' (single-source), and 'D' (disputed). This creates non-uniform ID namespaces in synthesized reports.

**Evidence:** bridge-commons section 'Output Schema'; debate-protocol Phase 2; deep-council Step 7

**Action:** Standardize all finding IDs to use bridge prefixes as the primary namespace, or use a unified 'F' prefix with bridge-origin metadata.

---

### G003 — [MEDIUM] Idempotency risk in artifact timestamps

**Domain:** async-queue
**Type:** finding
**Severity:** MEDIUM

**Description:** Artifact filenames use YYYYMMDD-HHMMSS which lacks sub-second precision. In high-concurrency environments (e.g., parallel-workflow triggering multiple councils), there is a significant risk of filename collisions if two artifacts are saved in the same second.

**Evidence:** bridge-commons section 'Artifact Format'; deep-council Step 8

**Action:** Append milliseconds to the timestamp or include the session_id/review_id at the start of the filename to ensure uniqueness.

---

### G004 — [HIGH] Terminology interchangeability in bridge_input mapping

**Domain:** content
**Type:** finding
**Severity:** HIGH

**Description:** deep-council maps both 'task_description' and 'context_summary' to the same '{context_summary}' value in Step 4. This collapses the distinction defined in bridge-commons between action-oriented instructions and passive context background, leading to redundant/confusing agent prompts.

**Evidence:** deep-council Step 4 'Bridge Input' JSON mapping; bridge-commons 'Input Schema' field definitions

**Action:** Update deep-council to map task_description to the specific intent/instruction and reserve context_summary for background data.

---

### G005 — [HIGH] Silenced Coverage via Auto-SKIPPED for HALTED Bridges

**Domain:** operations
**Type:** finding
**Severity:** HIGH

**Description:** In non-interactive mode, deep-council auto-SKIPS bridges that return HALTED (e.g., due to missing CLI auth). This silently drops coverage from critical model families without a non-zero exit code or failure state, potentially leading to a false sense of security in automated audits.

**Evidence:** deep-council Step 5 'HALTED Bridges' non-interactive default

**Action:** Change the non-interactive default to FAIL if a bridge is HALTED, unless the bridge is explicitly marked as optional in .bridge-settings.json.

---

### G006 — [MEDIUM] Inconsistent Verdict Thresholds

**Domain:** api
**Type:** finding
**Severity:** MEDIUM

**Description:** bridge-commons verdict logic (FAIL = any CRITICAL) differs from deep-council logic (FAIL = multi-model CRITICAL). While documented as intentional to account for depth asymmetry, it results in findings being 'downgraded' in severity during synthesis without adversarial challenge.

**Evidence:** bridge-commons 'Verdict Logic' vs deep-council 'Cross-Bridge Verdict' logic

**Action:** Align the definitions or provide a clear 'Mechanical Verdict' vs 'Council Verdict' distinction in the final report.

---

### G007 — [LOW] Ambiguous Timeout Scope for Directories

**Domain:** async-queue
**Type:** finding
**Severity:** LOW

**Description:** The timeout estimation table in bridge-commons uses file counts (e.g., 5-20 files). When a directory is provided as scope, it is undefined if the count is recursive or top-level. This can lead to significant under-estimation of runtime for deep directory structures.

**Evidence:** bridge-commons section 'Timeout Estimation'

**Action:** Clarify that the file count is recursive total or provide a tool-based file count pre-check before calculating timeout.

---

### G008 — [MEDIUM] Emoji usage violates no-emoji convention

**Domain:** content
**Type:** finding
**Severity:** MEDIUM

**Description:** parallel-workflow Step 6 uses emojis (checkmarks, Xs, skip arrows) in its progress and final reports. This violates the suite-wide convention for no-emoji artifacts, which are intended to be clean markdown for automated parsing and diffing.

**Evidence:** parallel-workflow section 'Step 6: Report Progress and Results'

**Action:** Replace emojis with standard markdown indicators or textual status labels (e.g., [OK], [FAIL]).

---

### G009 — [MEDIUM] TTL expiry logic allows stale execution state

**Domain:** operations
**Type:** finding
**Severity:** MEDIUM

**Description:** .bridge-settings.json TTL (24h) is only checked once during the discovery phase (Step 1B). For long-running or complex reviews, the settings could expire mid-execution, but the suite will continue using the cached (now stale) model discovery data.

**Evidence:** deep-council Step 1B 'Model Discovery & Settings' TTL check

**Action:** Move the TTL check to the dispatch phase or use a lockfile with an expiration timestamp to ensure fresh configuration.

---

### G010 — [HIGH] Context packet payload risk for stateless CLI bridges

**Domain:** async-queue
**Type:** finding
**Severity:** HIGH

**Description:** Stateless bridges (Gemini/Codex CLI) require embedding all previous round findings in Round N prompts. For large repositories or thorough intensity, the context packet size may exceed CLI character limits or model context windows, causing silent truncation or parse failures.

**Evidence:** bridge-commons section 'Context Passing Between Rounds'; bridge-gemini Execution section

**Action:** Implement finding summarization for Round N prompts or provide a file-based context injection path for CLI bridges.

---

### G011 — [MEDIUM] Undocumented workflow execution schema

**Domain:** content
**Type:** finding
**Severity:** MEDIUM

**Description:** parallel-workflow references 'schemas/workflow-execution-schema.json' for its output format, but this schema is not present in the repository or documented in the skill, preventing clients from validating workflow reports.

**Evidence:** parallel-workflow section 'Output Format'

**Action:** Create and document the workflow-execution-schema.json file in the parallel-workflow/schemas/ directory.

---

### G012 — [HIGH] Single point of failure in Council Debate Coordinator

**Domain:** operations
**Type:** finding
**Severity:** HIGH

**Description:** deep-council Step 6B relies on a single Debate Coordinator Task agent to perform adversarial validation. If this agent fails, times out, or returns invalid JSON, the entire council fails as there is no fallback to a mechanical synthesis or retry mechanism.

**Evidence:** deep-council Step 6 'Stage B: Debate — Adversarial Validation'

**Action:** Implement a fallback to mechanical synthesis (Step 6A results) if the Debate Coordinator fails, and mark the report as 'not-adversarially-validated'.

---

### G013 — [LOW] Redundant field aliases in bridge_input

**Domain:** api
**Type:** finding
**Severity:** LOW

**Description:** deep-council provides both session_id/review_id and scope/review_scope. This redundancy is intended for backward compatibility but increases the risk of 'split brain' logic in bridges where an adapter might read different values if they diverge.

**Evidence:** deep-council Step 4 'Bridge Input' JSON structure

**Action:** Deprecate the 'review_id/review_scope' aliases and enforce the canonical bridge-commons names (session_id/scope).

---

### G014 — [MEDIUM] Data loss: Omission of withdrawn_findings in Council report

**Domain:** api
**Type:** finding
**Severity:** MEDIUM

**Description:** bridge-commons requires recording 'withdrawn_outputs', but the final deep-council report JSON in Step 7 lacks this field. This results in the loss of audit history regarding which findings were invalidated during the council debate.

**Evidence:** deep-council Step 7 'Council Report Structure'; bridge-commons 'Output Schema'

**Action:** Add 'withdrawn_findings' and 'disputed_findings' arrays to the deep-council output schema to match bridge-level transparency.
