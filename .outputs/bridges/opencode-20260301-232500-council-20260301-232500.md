accordingly (Step 3A → Step 4A, Step 3B → Step 4B). Update the Bridge Identity section's
connection preference numbering note if it references step numbers by name.

### O-merged-002 — HIGH: Invalid Model Identifiers in .bridge-settings.json
**Severity:** HIGH
**Domain:** api / operations
**Confirmed by:** qwen3-max, MiniMax-M2.5, kimi-k2.5, glm-5 (4/4 models)
**intra_bridge_multi_model_confirmed:** true

**Description:** `.bridge-settings.json` contains placeholder model identifiers that do not
exist: `"gemini-3.1-pro-preview"` (Gemini models follow `gemini-{version}-{variant}` naming
such as `gemini-2.0-flash`, `gemini-1.5-pro`) and `"gpt-5.3-codex"` (GPT-5 does not exist;
Codex is deprecated). When bridge-gemini or bridge-codex reads these values and passes them
to dispatch, the call will fail. MiniMax rated this CRITICAL; qwen3, kimi, glm rated it HIGH.
Inheriting highest severity: **HIGH** (MiniMax's CRITICAL was context-specific to runtime
failures; other models assessed it as HIGH given bridges fall back to null/default).

**Evidence:** `.bridge-settings.json` lines 4–5:
```json
"gemini": { "enabled": true, "model": "gemini-3.1-pro-preview" },
"codex": { "enabled": true, "model": "gpt-5.3-codex" }
```

**Action:** Replace with valid identifiers (e.g., `"gemini-2.0-flash"` or `null` for default;
`"gpt-4o"` or `null` for codex). Setting to `null` is safest — bridges default to the runtime's
currently configured model.

### O-merged-003 — HIGH: Single Agent Performing Both DA and IC Roles
**Severity:** HIGH
**Domain:** async-queue / operations
**Confirmed by:** qwen3-max, MiniMax-M2.5, kimi-k2.5, glm-5 (4/4 models)
**intra_bridge_multi_model_confirmed:** true

**Description:** `deep-council/SKILL.md` Step 6 Stage B spawns a single "Debate Coordinator"
Task agent that simultaneously performs Devil's Advocate (DA) and Integration Checker (IC)
roles within the same prompt. This contradicts the design in `bridge-commons/SKILL.md` where
DA and IC are explicitly defined as separate parallel participants: "The total number of
parallel agents per round = `len(active_domains) + 2`" (one expert per domain, plus Challenger
and Integration Checker). A single agent cannot adversarially challenge its own conclusions.

**Evidence:**
- `deep-council/SKILL.md` lines 368–444: single prompt with "Role 1: Devil's Advocate" and
  "Role 2: Integration Checker"
- `bridge-commons/SKILL.md` lines 280–288: DA and IC defined as separate roles spawned in
  parallel at the start of each round

**Action:** Spawn two separate Task agents in Stage B: one for DA (challenges preliminary
findings) and one for IC (surfaces cross-component integration issues). Each agent receives
the same preliminary_findings context but operates independently to maintain adversarial rigor.

### O-merged-004 — MEDIUM: TTL Logic Operationally Unsound with Committed Timestamp
**Severity:** MEDIUM
**Domain:** operations
**Confirmed by:** qwen3-max, MiniMax-M2.5, kimi-k2.5(HIGH) (3/4 models — glm-5 assessed INFO)
**intra_bridge_multi_model_confirmed:** true (inheriting MEDIUM from 3-model majority)

**Description:** `deep-council/SKILL.md` Step 1B forces re-discovery when
`(current_time - updated) > ttl_hours`. The `updated` timestamp is embedded in `.bridge-settings.json`
as a committed value (`"2026-03-01T00:00:00Z"`). Since this file is version-controlled, any
working copy more than 24 hours old will always trigger re-discovery — defeating the caching
intent and creating unnecessary interactive prompts on every invocation. Note: kimi-k2.5
assessed this as HIGH with the additional observation that the file is git-tracked; glm-5
assessed as INFO (working as intended). Consensus severity: MEDIUM.

**Evidence:**
- `.bridge-settings.json` line 13: `"updated": "2026-03-01T00:00:00Z"` (static, committed)
- `deep-council/SKILL.md` line 198: TTL check logic

**Action:** Either (a) update the `updated` timestamp dynamically when settings are saved
during discovery runs, (b) use file mtime rather than embedded timestamp for TTL comparison,
or (c) separate the user-discoverable config from the TTL-tracked runtime state into two files
(e.g., `.bridge-settings.json` for config, `.bridge-settings-cache.json` for runtime state).

### O-merged-006 — MEDIUM: HALTED Semantics Inconsistent Between bridge-commons and deep-council
**Severity:** MEDIUM
**Domain:** api
**Confirmed by:** kimi-k2.5 (primary), confirmed direction by qwen3-max, glm-5 (3/4 models noted HALTED handling)
**intra_bridge_multi_model_confirmed:** true

**Description:** `bridge-commons/SKILL.md` defines HALTED as: "Requires user decision — no
provider configured, explicit user abort | Surface `halt_message` and wait; **do not continue**."
`deep-council/SKILL.md` Step 5 contradicts this: "Non-interactive default: auto-SKIPPED with
a warning in the report." These are conflicting instructions. A bridge implementing bridge-commons
would block; deep-council would auto-skip the same bridge. This creates divergent behavior
depending on whether the bridge or the council interprets the HALTED status.

**Evidence:**
- `bridge-commons/SKILL.md` lines 41–46: HALTED row — "do not continue"
- `deep-council/SKILL.md` lines 322–328: auto-SKIPPED with warning in non-interactive mode

**Action:** Reconcile the semantics. Option A: bridge-commons acknowledges that orchestrators
may override HALTED in non-interactive mode — add a note to the HALTED row. Option B:
deep-council removes the auto-skip override and surfaces the halt message to the user.
Document which takes precedence.
